# X.509 certifikát

**X.509** je standard ITU-T pro certifikáty veřejných klíčů. Definuje *strukturu* dat, která spojuje veřejný klíč s identitou (subjektem), podepsanou důvěryhodnou stranou (issuerem). Internetová podoba je v RFC 5280 (2008).

Verze 3 (1996) je dnešní standard, použitý ve všech TLS, S/MIME, code signing, EMV chip cards.

## Struktura certifikátu

X.509 certifikát je struktura ASN.1 (Abstract Syntax Notation One), kódovaná DER (Distinguished Encoding Rules) nebo PEM (base64-encoded DER s hlavičkami).

### Hlavní pole

| Pole | Význam |
| :--- | :--- |
| `version` | v1 = 0, v2 = 1, v3 = 2 (povinně v3 dnes) |
| `serialNumber` | Unikátní číslo v rámci issueru. Pro veřejné CA musí být *nepředvídatelné* (RFC 5280) — min. 64 bitů entropie. |
| `signatureAlgorithm` | OID podpisového algoritmu (např. `1.2.840.113549.1.1.11` = `sha256WithRSAEncryption`) |
| `issuer` | DN (Distinguished Name) CA, která certifikát vydala |
| `validity` | `notBefore` a `notAfter` — období platnosti |
| `subject` | DN majitele veřejného klíče |
| `subjectPublicKeyInfo` | Algorithm ID + samotný veřejný klíč |
| `extensions` | Volitelná rozšíření (v3 only) — viz dále |

### Distinguished Name (DN)

X.500 hierarchická adresa, např.:

```
CN = example.com
O  = Example Corp.
L  = San Francisco
ST = California
C  = US
```

Pro TLS certifikáty se moderně **CN ignoruje pro hostname match** (nepoužívá se), a používá se **Subject Alternative Name (SAN)** extension.

## Extensions — klíčové části V3

Extension je trojice `(OID, critical, value)`. Pokud `critical = TRUE` a klient nezná tu extension, *musí* certifikát odmítnout.

### Klíčové extensions

#### Subject Alternative Name (SAN)

Seznam DNS jmen, IP adres, e-mailů, URI pro které certifikát platí.

```
SAN:
  DNS: example.com
  DNS: www.example.com
  DNS: *.cdn.example.com
  IP: 192.0.2.10
```

Wildcard (`*.example.com`) pokrývá jednu úroveň subdomény, *ne* hlubší (`x.y.example.com` neodpovídá).

> **CN je deprecated pro hostname.** Chrome 58+ a všechny moderní klienty vyžadují SAN. Pokud certifikát nemá SAN, hostname check selže i když CN odpovídá.

#### Basic Constraints

```
basicConstraints:
  cA: TRUE              ← CA certifikát
  pathLenConstraint: 0  ← max hloubka pod-CA
```

`cA: TRUE` označuje certifikát jako CA (může podepisovat další certifikáty). Pro end-entity certifikáty *musí* být `cA: FALSE` nebo extension chybí.

> **Pozor:** historicky chybějící basicConstraints v end-entity certifikátech *byla* využitelná — útočník použil svůj cert jako CA (Marlinspike 2009). Dnes všechny prohlížeče vyžadují *explicitní* basicConstraints.

#### Key Usage

Bitmaska povolených operací:

```
keyUsage:
  digitalSignature
  keyEncipherment
  dataEncipherment
  keyAgreement
  keyCertSign     ← pouze pro CA
  cRLSign         ← pouze pro CA
```

TLS server typicky: `digitalSignature, keyEncipherment` (RSA) nebo `digitalSignature, keyAgreement` (ECC).

#### Extended Key Usage (EKU)

OID seznam specifických cílů:

* `1.3.6.1.5.5.7.3.1` — `id-kp-serverAuth` (TLS server)
* `1.3.6.1.5.5.7.3.2` — `id-kp-clientAuth` (TLS client)
* `1.3.6.1.5.5.7.3.3` — `id-kp-codeSigning` (signed code)
* `1.3.6.1.5.5.7.3.4` — `id-kp-emailProtection` (S/MIME)
* `1.3.6.1.5.5.7.3.8` — `id-kp-timeStamping` (TSA)

#### CRL Distribution Points (CRLDP)

URL CRL souboru pro revokační check. Viz [[revokace]].

```
cRLDistributionPoints:
  http://crl.example-ca.com/intermediate.crl
```

#### Authority Information Access (AIA)

URLs pro:

* **CA Issuers** — fetch intermediate CA certifikát.
* **OCSP** — endpoint pro online status check. Viz [[revokace]].

```
authorityInfoAccess:
  caIssuers: http://ca.example.com/intermediate.crt
  OCSP: http://ocsp.example-ca.com
```

#### Authority Key Identifier (AKI), Subject Key Identifier (SKI)

Hash veřejného klíče — slouží k *rychlé identifikaci* v chain building. SKI je hash certifikátova VK, AKI je hash issuerova VK.

#### Signed Certificate Timestamp (SCT)

Důkaz, že certifikát byl publikován v Certificate Transparency logu (RFC 6962). Vyžadováno všemi moderními prohlížeči pro nové certifikáty.

## Chain validation

Klient ověřuje certifikační řetězec:

1. **Server cert** — ověř podpis pomocí Intermediate CA $VK$.
2. **Intermediate cert** — ověř podpis pomocí Root CA $VK$ z trust store.
3. **Root cert** — je *trusted anchor*, nepotřebuje další ověření.

Pro každý certifikát v chainu kontroluj:

* Podpis platný.
* Validity period zahrnuje současný čas.
* `basicConstraints.cA = TRUE` pro non-leaf certs.
* Path length constraint respektovaný.
* Key usage konzistentní (CA → `keyCertSign`).
* Hostname matching (pro leaf cert): SAN obsahuje cílovou doménu.
* Certifikát *není revokován* (CRL/OCSP, [[revokace]]).
* Algorithm constraints (žádné slabé hashe MD5/SHA-1).

## Velikost klíče a algorithm constraints

Současná doporučení (NIST SP 800-57, CA/Browser Forum):

| Algoritmus | Minimum |
| :--- | :--- |
| RSA | 2048 bitů (povinné od 2014) |
| RSA root | 4096 bitů (nově vydávané) |
| ECDSA | P-256 (256 bitů) |
| Signature hash | SHA-256 (SHA-1 zakázáno od 2017, MD5 od 2009) |
| EdDSA | Ed25519 (Chrome plánuje support; není standardní v PKI) |

Slabé certifikáty (RSA-1024, SHA-1) jsou *odmítnuty* všemi moderními prohlížeči.

## Self-signed certifikáty

Certifikát, kde `issuer = subject` a podpis vlastním klíčem. Použití:

* **Root CA** — kořen důvěry.
* **Vývoj / testing** — `mkcert`, OpenSSL self-signed.
* **Embedded zařízení** — kamery, NAS s vlastním HTTPS bez veřejné CA.

Prohlížeč varuje "Your connection is not private" — uživatel musí explicitně schválit.

### mkcert

[mkcert](https://github.com/FiloSottile/mkcert) — vytvoří lokální CA a podepíše dev certifikáty. CA je *přidán do trust store* lokálně, takže prohlížeč nezobrazí varování. Pohodlné pro vývojáře.

## Formáty kontejnerů

* **PEM** (Privacy Enhanced Mail) — base64 + hlavičky. Nejběžnější.
* **DER** (Distinguished Encoding Rules) — binární. Java truststores.
* **PKCS#7** (.p7b, .p7c) — kontejner pro certifikáty bez private key. Často "certifikační chain".
* **PKCS#12** (.p12, .pfx) — kontejner se certifikátem *i* privátním klíčem, šifrovaný heslem. Windows export.

## SPKI / pinning

Hash veřejného klíče (Subject Public Key Info hash, SPKI hash) lze "pinovat" v aplikaci — aplikace odmítne komunikaci, pokud server prezentuje *jiný* veřejný klíč.

* **HPKP** byl protokol pro web — deprecated 2017.
* **Mobile / native apps** — *certificate pinning* je stále best practice (Facebook, Twitter, banking apps).
* **CT** + CAA částečně nahrazují pinning na webu.

## Praktický check certifikátu

```bash
# Načtení a parsování
openssl x509 -in cert.pem -noout -text

# Verifikace s chain
openssl verify -CAfile root.pem -untrusted intermediate.pem cert.pem

# TLS connect a získání chain
openssl s_client -showcerts -connect example.com:443 < /dev/null

# CT verification
ct-go check-cert cert.pem    # SCTs are valid

# Online tester
https://www.ssllabs.com/ssltest/
```
